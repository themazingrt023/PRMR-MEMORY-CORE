# V0.79 Controlled Hosted Test Scope

V0.79 adds a synthetic controlled hosted test scope so the Render backend can be smoke-tested through real hosted HTTP protected routes.

This is controlled synthetic hosted smoke evidence only. It is not production access, not real client onboarding, not billing, not external validation, not bank approval, not compliance approval, not legal approval, not external security certification, and not real-world validation.

## Hosted Backend

- Backend URL: `https://prmr-memory-core-api.onrender.com`
- FastAPI entrypoint: `prmr.product.api_server_v076:app`
- Hosted start command: `uvicorn prmr.product.api_server_v076:app --host 0.0.0.0 --port $PORT`

## Render Environment Variables

Set these in Render before attempting the full protected hosted smoke:

```text
PRMR_ENABLE_CONTROLLED_TEST_SCOPE=true
PRMR_TEST_API_KEY=<fresh synthetic test key generated by the founder/operator>
PRMR_TEST_CLIENT_ID=client_demo
PRMR_TEST_VAULT_ID=vault_demo
PRMR_TEST_NAMESPACE=default
PRMR_ALLOWED_ORIGINS=https://prmr-memory-core.vercel.app,http://localhost:3000
PRMR_SYNTHETIC_ONLY=true
```

Rules:

- Generate a fresh synthetic test key outside the repo.
- Do not commit the test key.
- Do not paste the full key into docs, reports, screenshots, or public messages.
- Do not reuse old local/dev keys.
- Rotate or delete this test key after the smoke window if it is no longer needed.

## Local Smoke Environment

Set matching local variables when running the smoke from this workspace:

```powershell
$env:PRMR_HOSTED_API_URL="https://prmr-memory-core-api.onrender.com"
$env:PRMR_TEST_API_KEY="<same fresh synthetic test key configured on Render>"
$env:PRMR_TEST_CLIENT_ID="client_demo"
$env:PRMR_TEST_VAULT_ID="vault_demo"
$env:PRMR_TEST_NAMESPACE="default"
```

Do not put the raw value in `.env.example`, reports, or tracked files.

## What The Server Does

When `PRMR_ENABLE_CONTROLLED_TEST_SCOPE=true`, the hosted FastAPI process registers one synthetic client, vault, namespace, and active key record from environment variables.

The raw key is read from environment only. The server stores/uses a hash for validation and exposes only a safe preview/hash-prefix status in health evidence.

If the enable flag is false, the scope is disabled.

If the enable flag is true but required variables are missing, `/health` reports `NEEDS_TEST_SCOPE_CONFIG` and the full smoke must not claim a pass.

## Commands

Readiness audit:

```powershell
python examples/audit_v079_controlled_hosted_test_scope.py
```

Full protected hosted smoke after Render and local test-scope variables are set:

```powershell
python examples/run_full_hosted_protected_smoke_v079.py
```

Expected result levels:

- `NEEDS_HOSTED_URL`: local hosted URL variable is missing.
- `NEEDS_TEST_SCOPE`: hosted URL exists but local test-scope variables are missing.
- `NEEDS_TEST_SCOPE_CONFIG`: test scope was requested but required variables are incomplete.
- `PASS_FULL_CONTROLLED_HOSTED_SMOKE`: all protected hosted flow and blocked-flow checks passed.
- `NEEDS_WORK`: a required hosted smoke check failed.

## Protected Flow Covered

- `POST /v1/events/ingest`
- `POST /v1/continuity/packet`
- `POST /v1/memory/reconstruct`
- `POST /v1/explain`
- `POST /v1/actions/least-harm`
- `GET /v1/reports/{report_id}`
- `GET /v1/usage`
- `GET /v1/dashboard/state`

Blocked-flow checks:

- wrong key is blocked
- wrong vault is blocked
- wrong namespace is blocked
- missing Authorization is blocked
- malformed Authorization is blocked

## Boundary

V0.79 only proves a controlled synthetic hosted test scope when the protected smoke passes against the real hosted URL. It does not grant live customer access, hosted onboarding, billing, production readiness, external validation, bank approval, compliance approval, legal approval, external security certification, or real-world validation.
